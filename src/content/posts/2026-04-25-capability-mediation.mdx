*A field report from building a secure MCP gateway over private state.*

This is an experiment report, not a finished system.

The thing it describes is running on a Mac mini under my desk and serves a handful of personal agents I trust enough to read selected financial signals, GitHub activity, research feeds, and market calendars — but not enough to hand a credit card to.

It is in flight. The rough edges are visible from orbit. The parts I would ship to someone else are smaller than the parts I would rewrite.

I am writing now, while it is still in flight, because one design choice has turned out to matter more than I expected:

> Mediate capabilities, not tools.

That sentence sounds abstract, but the practical difference is sharp.

A tool says: "Here is an API. Call it correctly."

A capability says: "Here is a bounded kind of action you are allowed to perform, over a bounded kind of state, with observable behavior, revocable access, and a uniform contract."

That shift changed how the system felt to build, debug, and trust.

---

## Why we built it

The lived problem is small and concrete.

I have personal data that I would love an agent to actually use: brokerage signals, earnings calendars, FOMC dates, GitHub repos I follow, public filings, research papers from Anthropic and OpenAI, selected timelines and newsletters, and a few local knowledge stores.

None of that data is trivial.

Some of it is public but noisy. Some of it is private but useful. Some of it is stale unless refreshed. Some of it should be readable but never writable. Some of it should be discoverable only if the agent already has the right scope.

The obvious approach is to give agents tools — one per provider, one per API. That works for demos. It degrades in real use.

Every new tool brings a new schema, a new auth story, a new error surface, a new freshness model, and a new way for agents to hallucinate what the tool can do. The agent ends up reasoning about API topology as much as it reasons about the user's actual task.

That is the wrong cognitive load.

The experiment was to put a narrow gateway between agents and private state. Not a tool zoo. Not a god-mode backend. A capability-mediated layer.

The agent should not need to know whether a fact came from OpenBB, GitHub, a local SQLite index, or a calendar feed. It should know what source exists, what capabilities are allowed, how fresh the result is, how to resolve an item, and whether a source is unavailable because of missing credentials or a real failure.

The gateway owns the messy part. The agent owns the planning.

## The core design

The system is built around a uniform `Item` envelope.

Every source returns items with the same basic shape:

```ts
type Item = {
  source: string
  source_id: string
  kind: string
  title?: string
  url?: string
  uri?: string                 // server-populated
  created_at: string
  seen_at: string
  tags: string[]
  content: Record<string, unknown>
}
```

The exact content fields vary by source, but the contract does not.

A GitHub issue, an earnings event, a Form 4 filing, a research paper, a social bookmark, a calendar event, or a gateway health record all move through the same retrieval surface.

The public capability surface is intentionally small:

```text
recent(source, n, since?, until?)
search(query, source?, since?)
resolve(uri)
fresh_pull(source)
resources/read(source://<name>/<id>)
resources/subscribe(source://<name>/<id>)
```

Agents discover sources through `gateway.health`, then use the same retrieval grammar everywhere else.

```text
recent("gateway.health", n=100)     # what is available right now?
recent("fin.fomc", n=10)            # upcoming FOMC meetings
recent("fin.earnings", n=20)        # upcoming earnings window
recent("fin.insider", n=10)         # recent Form 4 filings
recent("x.watched", n=30)           # latest posts from watched accounts
```

The important part is not that these are clever tools. They are not.

The important part is that the tools are boring, uniform, and capability-scoped.

A source can be live, stubbed, stale, or missing credentials without becoming an exception-shaped surprise. The health record looks like:

```json
{
  "source": "your.broker.balance",
  "available": false,
  "reason": "credential 'env:YOUR_API_KEY' not found"
}
```

That means the agent sees "configure me" instead of "the world exploded."

For personal agents, broken partial state is the normal state. The system has to make partial availability legible.

## What we actually tested

I ran five real cross-source queries against the live gateway. The first was the one that motivated this whole post; the next four push harder by mixing personal cash position, calendar primitives, and social signals.

### Case 1: the original query

Late one night, working through a position-sizing question, I asked:

> *"I have $20k cash. Looking at the next 14 days of earnings plus the FOMC meeting on the 29th, when's the lowest-vol-risk week to enter a new position?"*

The agent composed an answer from `fin.fomc`, `fin.earnings`, and `fin.insider` — all reachable through the same tools, the same `Item` shape, and the same scope grammar.

It returned a windowed calendar:

```text
Apr 29 (Wed)   FOMC + MSFT/GOOGL/AMZN/META   HARD BLOCK — vol cluster
Apr 30 (Thu)   AAPL earnings                 HARD BLOCK
May 1 (Fri)                                  CLEAR
May 5 (Tue)    AMD earnings                  HARD BLOCK
May 6 (Wed)                                  CLEAR — first fully clean day
May 7–8                                      CLEAR through May 12
```

Recommendation: enter May 6–8, after FOMC and the surrounding mega-cap earnings cluster clear.

This is the kind of result I wanted the system to make possible: not a bot calling a finance API, and not a single-purpose workflow hardcoded around one provider, but an agent composing private state, public market calendars, and bounded access into one answer.

### Cases 2–5: harder cross-source compositions

The four follow-on tests added social signals to the mix — the hardest case for the `Item` envelope, because tweet JSON and Form 4 XML have almost nothing structurally in common.

**Case 2 — Personal position + watched X accounts + FOMC sentiment.** *"What are the people I follow on X actually saying about the Fed right now?"* The agent composed `x.watched` + `fin.fomc` + `fin.earnings`. The interesting finding was the *absence* of signal: of 22 recent posts from watched accounts, nobody was talking about the Fed at all. That silence is itself a read. Verdict: wait, deploy at most $5k now; preserve cash optionality through the Apr 29 vol cluster.

**Case 3 — Bookmarks + insider buying.** *"Of the stocks I've been bookmarking on X, which ones also have officer/director buying?"* The agent composed `x.bookmarks` + `fin.insider`, with an explicit `search("BUY", source="fin.insider")` returning `[]`. **Double null:** zero tradeable tickers in 30 bookmarks, zero buy-side insider filings in 25 Form 4s. Verdict: hold the capital. The clean negative confirmation matters — in a tool-zoo agent, "couldn't find anything" is hard to distinguish from "didn't look hard enough."

**Case 4 — Research catalyst timing.** *"Where's the cleanest 3-day window to deploy on AI-momentum without overlapping a vol event?"* The agent composed `anthropic.research` + `openai.research` + `github.releases` + `fin.fomc` + `fin.earnings`. Same May 6–8 answer as case 1, by a different reasoning path. Independent convergence is a small validation.

**Case 5 — Cross-platform thesis surfacing.** *"What thesis am I implicitly building from a month of X and Xiaohongshu bookmarks?"* The agent composed `x.bookmarks` + `xhs.bookmarks` + `fin.insider` + `fin.filings`. The honest read: about 70% of my bookmarks are agent-tooling and meta-engineering. The bookmarks describe a *builder* thesis, not an *investor* thesis. Forcing a mapping onto GOOGL/META/AMZN would be fabrication, especially with insiders selling those names. Verdict: hold the capital.

Three of five cases declined to fully deploy the $20k (case 2 partial, cases 3 and 5 full hold). To be unambiguous: that is the agent declining to spend, not the agent declining to trust the gateway. The gateway is what made the negative results legible in the first place.

### What it cost vs. raw tools

I tracked tool calls and approximate input tokens for each query (counted from the gateway's actual response payloads), then estimated the native equivalent by summing typical raw provider payload sizes for the same data — uncapped X tweet objects with expanded entities, full SEC EDGAR Form 4 XML, full Anthropic/OpenAI blog HTML, OpenBB earnings dicts, plus the per-client tool-definition footprint each one would carry in agent context. The native bounds widen when raw payloads are larger than expected.

| # | Test case | Calls | Gateway tokens | Native tokens (est.) | Reduction |
|---|---|---:|---:|---:|---:|
| 1 | Cash + earnings + FOMC | 4 | ~12,500 | ~85,000–100,000 | ~7× |
| 2 | Cash + watched X + FOMC | 4 | ~12,000 | ~80,000–150,000 | ~7–12× |
| 3 | Cash + bookmarks + insider | 4 | ~22,000 | ~88,000–132,000 | ~4–6× |
| 4 | Cash + research + calendar | 6 | ~22,000 | ~88,000–110,000 | ~4–5× |
| 5 | Cash + cross-platform thesis | 5 | ~28,000 | ~85,000–140,000 | ~3–5× |

Twenty-three calls across five questions. **~96,500** input tokens through the gateway versus an estimated **~430,000–630,000** native — roughly **5–7× fewer agent-context tokens for the same answer**.

That is the agent-side cost. The gateway doesn't make data appear from nowhere — someone has to fetch from the providers. The shift is moving that cost out of the agent's context window (where every token is paid for in inference cost, repeated per query) and into a cron run (where it's amortized across every read of the snapshot).

The savings come from three layers:

1. **Content normalization.** Anthropic blog posts go from 15–25KB raw HTML to ~600 tokens after a 2000-char per-field cap. A Form 4 XML filing goes from 5–8KB to ~600 tokens. Oversize dicts (full GitHub release API responses) get replaced with `{_truncated: true, _original_size_chars: N}` stubs.
2. **Schema collapse.** Five distinct provider schemas — X tweet JSON, EDGAR Form 4 XML, OpenBB earnings dicts, Anthropic blog HTML, FOMC calendar — become one `Item` shape. Tool definitions in agent context drop from ~1,500–6,000 tokens to ~250.
3. **Snapshot serving.** Most reads come from a pre-built `signals.jsonl` written once by an upstream cron, not live API calls. The agent doesn't pay provider rate limits or paid-tier per-call costs.

That last layer matters most for sources behind paid or rate-limited APIs. An agent doing 20 cross-source queries a day would burn most of an X API Basic tier ($200/mo) hitting bookmark and timeline endpoints directly. Through the gateway, the per-query API cost is ~0 — the cron pulls every 15 min and serves all subsequent reads for free.

The gateway pays a small fixed cost (cron CPU, ~250 tokens of tool definition, snapshot storage) to amortize a large variable cost (per-query API hits, schema parsing, raw payload tokens). The break-even point is around the second cross-source query in a session.

## Why not just expose tools?

The standard answer is: "Just give the agent tools and let it decide."

I tried that. It works for demos. It degrades in real use.

The problem is not that agents cannot call tools. They can.

The problem is that tool exposure shifts too much trust to the least stable layer of the system.

With raw tools, the agent has to infer:

- which APIs exist
- which credentials are configured
- which sources are stale
- which calls are safe
- which calls mutate state
- which errors are retryable
- which data shapes are equivalent
- which result is canonical
- which source should be trusted more

That is a lot of implicit contract.

Capability mediation moves those assumptions into infrastructure.

Instead of exposing "brokerage API," the gateway exposes bounded financial read capabilities.

Instead of exposing "GitHub token," it exposes scoped repository activity.

Instead of exposing "database query," it exposes source-specific retrieval over a uniform item shape.

Instead of letting an agent discover failure through stack traces, it gives the agent a health and availability surface.

The distinction is subtle but important:

> A tool is an affordance.  
> A capability is an affordance plus boundary, state, policy, and observability.

For personal agents, that boundary matters.

## The scope grammar

The system uses scope strings to control what an agent can see or do.

The exact grammar is still evolving, but the idea is simple:

```text
read:source.fin.earnings
read:source.fin.fomc
read:source.fin.insider
read:source.x.watched
read:source.gateway.health
subscribe:source.fin.insider
```

Wildcards are segment-boundary only. `read:source.fin.*` is valid. `read:source.fin*` is not — that would over-grant against `source.finch.x`.

Scopes are not just auth decorations. They are part of the agent contract.

A source may exist, but not be visible to an agent.

A resource may be resolvable, but not subscribable.

A source may allow `recent` but not `fresh_pull`.

The gateway can answer:

```text
You may read this.
You may not refresh this.
This exists but is not configured.
This source is stubbed.
This source is live.
This source is stale.
```

That gives the agent something much better than binary success/failure.

It gives it a map of its own permissions.

That matters because agents are very good at proceeding confidently when they should instead ask, refresh, degrade, or stop.

## What surprised me

The first surprise was how much value came from boring uniformity.

I expected the hard part to be adding sources.

It was not.

The hard part was making sources feel the same enough that agents could compose them without custom glue.

Once everything looked like an `Item`, the agent stopped caring where things came from. That was the point.

The second surprise was that `gateway.health` became a first-class source.

Originally, health was for me. A debugging endpoint. Something I would inspect when the system broke.

But agents used it as part of planning.

They would check which sources were available, notice missing credentials, detect newly registered sources, and route around unavailable ones.

That changed my view of observability.

For agentic systems, observability is not only for humans after failure. It is runtime context for the agent before and during action.

The third surprise was that "stubbed" sources are better than absent sources.

A missing source makes the agent guess.

A stubbed source lets the system say:

```text
This source is conceptually part of the world, but it is not configured yet.
```

That is much easier for both the agent and the human operator to reason about.

It turns absence into state.

The fourth surprise was the cost gap. I expected the gateway to be cleaner. I did not expect it to be 5–7× cheaper in agent-context tokens — and effectively free on the external-API side, because the cron amortizes that cost across every read.

## What this is not

This is not a trading system.

This is not an autonomous financial advisor.

This is not a claim that agents should be allowed to execute high-risk actions just because they can read high-value state.

The current system is intentionally biased toward read, summarize, compare, and explain.

It is not optimized for autonomous execution.

That distinction matters.

A personal agent that can read private context is already useful. A personal agent that can mutate private state needs a much harder control plane: approvals, budgets, audit trails, rollback semantics, rate limits, and probably separate execution identities.

Capability mediation is a prerequisite for that world, not the whole answer.

## The security shape

The gateway is not secure because it is small.

It is secure because the sensitive operations are made explicit.

The design tries to avoid giving agents broad ambient authority.

Instead of "here is a token," the agent receives scoped access to specific capabilities.

Instead of "here is a database," the agent receives `recent`, `search`, `resolve`, and `fresh_pull`.

Instead of "here is every account," the agent receives only sources its scope allows.

Instead of "call anything," it gets a small grammar.

This is not enough for a production system. But it points in the right direction.

The security model I want is less like:

> "Can this agent use the brokerage tool?"

And more like:

> "Can this agent read position summaries from this source, resolve historical filing items, and subscribe to future filing updates — but not place orders, move money, or reveal raw credentials?"

That is the level where personal agent permissions start to feel real.

## Why MCP is a good fit

MCP is useful here because it gives the gateway a standard way to expose tools and resources without making every source a bespoke integration.

But the important part is not "we used MCP."

The important part is what MCP makes natural:

- tools for bounded actions
- resources for addressable state
- subscriptions for updates
- a server boundary between agent and data
- a place to enforce policy
- a place to normalize source behavior

The gateway is not just an adapter. It is a mediation layer.

That is the difference.

An adapter says:

> "Here is the API in another shape."

A mediation layer says:

> "Here is the capability you are allowed to exercise."

## The bigger pattern

This project has made me think that a lot of personal-agent infrastructure should be built around three surfaces:

1. **Capability surface** — what the agent can do.
2. **State surface** — what the agent can observe.
3. **Diagnostic surface** — what the agent can know about the system itself.

Most systems expose the first one badly, the second one inconsistently, and the third one only to humans.

Agents need all three.

They need to know not just the task, but the operating condition.

They need to know what is fresh, what is stale, what is blocked, what is stubbed, what is live, and what they are allowed to touch.

Otherwise, every agent becomes a confident intern wandering through a dark server room with a root token.

That is not the future I want.

---

## What comes next

The next version needs better policy and better memory.

Policy, because scopes need to become more expressive. Some capabilities should be time-bound. Some should require approval above a threshold. Some should be read-only except during explicitly delegated sessions. Some should expose summaries but not raw records.

Memory, because agents should remember source behavior over time. Not just "this source exists," but "this source is often stale," "this source usually needs refresh," "this source is safe for planning but not execution," and "this source failed yesterday because credentials were missing."

The gateway should also expose better provenance.

When an agent answers from three sources, the answer should carry source lineage naturally:

```text
This conclusion came from:
- FOMC calendar
- earnings window
- insider filing availability
- gateway health state
```

Not as an afterthought. As part of the result.

Eventually, I want agents to operate with something like a personal capability graph: a live map of what they can observe, what they can do, what requires approval, and what is currently broken.

That feels like the right abstraction.

Not tools.

Not plugins.

Not one giant assistant with everything wired in.

A mediated capability layer between agent reasoning and personal state.

---

## Closing thought

The most interesting part of this experiment is not that an agent answered a finance question.

It is that the answer emerged from a small, scoped, observable capability surface instead of a pile of bespoke tools.

The gateway did not make the agent smarter.

It made the world less chaotic.

That may be the real job of personal-agent infrastructure: not to replace reasoning, but to shape the environment so reasoning has something stable to stand on.
